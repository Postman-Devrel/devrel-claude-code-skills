# Inclusive Language

When writing for the Learning Center, avoid words that are not inclusive, and instead use more inclusive language to ensure a welcoming environment for all of our users.

## Terminology

The table below lists some commonly-used terms that are not inclusive and offers more inclusive alternatives.

| Don't use | Use instead |
|-----------|-------------|
| abort | end, close, stop, cancel |
| agnostic | platform-independent |
| blacklist, whitelist | blocklist, allowlist |
| disable, disabled | turn off, deselect, inactive, unavailable, deactivated |
| discrimination, discriminator | condition, property, test |
| hangs | stops responding, stops, closes |
| master, slave | primary/secondary, main/branch |
| normal | expected, typical |
| dummy | sample |
| old | Don't use old/older/newer. Use earlier, later, or latest. |
| orphan | *(see footnote [15])* |
| see | view — ("to learn more, see some_topic" is okay) |
| segregate | divide, isolate, quarantine, sequester, insulate |
| white-box, black-box | For "white-box testing" use "transparent-box testing," "clear-box testing," or "structural testing." For "black-box testing" use "opaque-box testing" or "specification-based testing." |
| widow | *(see footnote [15])* |

## Gender and pronouns

Avoid gendered words, including pronouns, but feel free to use "they" and "them," and to address the user as "you." Choose pronouns carefully, especially in cases where it may not be clear what you're referring to. Relative pronouns (such as "that" and "they") can help reduce ambiguity. Be mindful of pronoun choice when referring to the Postman user's users, for example a consumer of an API that a Postman user is publishing.

For more information, see [Inclusive Language Guide for Tech Companies and Startups](https://medium.com/pm101/inclusive-language-guide-for-tech-companies-and-startups-f5b254d4a5b7).
